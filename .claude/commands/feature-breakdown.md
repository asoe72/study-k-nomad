# Problem Breakdown

주어진 문제를 분석하고 실행 가능한 서브태스크로 세분화합니다.

## 해결해야 할 문제

$ARGUMENTS

## 작업 절차

### Step 1 — 문제 분석

아래 항목을 파악합니다:

- **핵심 목표**: 이 문제를 해결하면 무엇이 달성되는가?
- **제약 조건**: 기술 스택, 수정 불가 영역, 범위 제한
- **완료 기준**: 어떤 상태가 되면 문제가 해결된 것인가?
- **관련 파일/모듈**: 영향을 받는 코드 영역 (필요 시 Glob/Grep으로 탐색)

### Step 2 — 분해 전략 선택

문제 특성에 맞는 분해 기준을 선택합니다:

| 전략 | 적합한 상황 |
|------|------------|
| 레이어별 | UI / 로직 / 데이터가 명확히 분리되는 경우 |
| 피처별 | 독립적인 기능 단위로 나뉘는 경우 |
| 페이즈별 | 순서가 중요한 단계적 작업 |
| 의존성 기준 | 선행 완료가 필요한 작업이 있는 경우 |

### Step 3 — 서브태스크 세분화

다음 기준을 모두 만족하도록 태스크를 나눕니다:

- **독립성**: 다른 태스크 없이 단독으로 작업 가능한가?
- **검증 가능성**: 완료 여부를 명확히 확인할 수 있는가?
- **적정 크기**: 한 번의 집중 작업으로 완료 가능한 크기인가?

각 태스크에는 다음 정보를 포함합니다:
- 제목 (imperative form: "~를 구현한다")
- 전체 목표에서 이 태스크의 역할
- 관련 파일 또는 인터페이스
- 완료 기준

### Step 4 — 의존성 매핑

- 병렬 실행 가능한 태스크 그룹 식별
- 선행 완료가 필요한 태스크 → 후행 태스크 순서 명시
- 블로커가 되는 태스크 강조

### Step 5 — TaskCreate로 등록

세분화된 모든 태스크를 TaskCreate 툴로 등록합니다.
의존 관계가 있는 경우 TaskUpdate로 `addBlockedBy`를 설정합니다.

### Step 6 — 검증

등록 후 아래 항목을 확인합니다:

- [ ] 전체 목표와 서브태스크의 합이 일치하는가?
- [ ] 빠진 태스크는 없는가?
- [ ] 의존성 순서가 올바른가?
- [ ] 병렬 처리 가능한 태스크를 최대화했는가?

검증 결과를 사용자에게 요약하여 보고합니다.

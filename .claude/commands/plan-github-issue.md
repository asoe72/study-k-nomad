# Plan GitHub Issue

GitHub 이슈를 가져와 해결 계획을 수립하고 사용자 승인을 받습니다.

## 이슈 번호

$ARGUMENTS

## 작업 절차

### Step 1 — 이슈 정보 수집

아래 명령어로 이슈 전체 내용을 가져옵니다:

```bash
gh issue view $ARGUMENTS --json number,title,body,labels,assignees,comments,state
```

이슈가 존재하지 않거나 접근 불가한 경우 즉시 사용자에게 알리고 중단합니다.

### Step 2 — 코드베이스 탐색

이슈 내용을 바탕으로 관련 코드를 파악합니다:

- 이슈에 언급된 파일, 함수, 컴포넌트, 에러 메시지 등을 Glob/Grep으로 검색
- 영향 범위(수정이 필요한 파일 목록) 파악
- 관련 테스트 파일 존재 여부 확인
- 현재 구현 방식과 이슈가 요구하는 상태 간의 차이(gap) 식별

### Step 3 — 이슈 분석

수집한 정보를 바탕으로 아래 항목을 정리합니다:

- **이슈 요약**: 무엇이 문제이고 무엇을 해결해야 하는가
- **근본 원인**: 버그라면 어디서 왜 발생하는지, 기능 요청이라면 현재 무엇이 없는지
- **영향 범위**: 수정 시 영향받는 파일/모듈/기능
- **완료 기준**: 이슈가 해결된 상태를 어떻게 확인할 수 있는가
- **위험 요소**: 주의해야 할 side-effect나 breaking change

### Step 4 — 구현 계획 수립

다음 형식으로 단계별 실행 계획을 작성합니다:

```
## 구현 계획

### Phase 1 — [단계명]
- [ ] 작업 1: [파일명:함수명] — [구체적인 변경 내용]
- [ ] 작업 2: ...

### Phase 2 — [단계명]
- [ ] 작업 3: ...

## 병렬 처리 가능한 작업
- Phase X의 작업 A와 작업 B는 독립적으로 동시 진행 가능

## 예상 변경 파일 목록
- `경로/파일명` — 변경 이유
- ...
```

각 작업에는 반드시 다음을 포함합니다:
- 대상 파일과 위치 (가능하면 `파일경로:라인번호`)
- 구체적인 변경 내용 (무엇을 추가/수정/삭제할지)
- 해당 작업의 완료 기준

### Step 5 — 계획 요약 보고 및 승인 요청

사용자에게 다음 내용을 보고합니다:

1. **이슈 요약** — 제목, 번호, 유형
2. **분석 결과** — 근본 원인 또는 구현 방향
3. **구현 계획** — Phase별 작업 목록
4. **변경 파일 목록** — 예상 수정 범위

보고 후 반드시 `ExitPlanMode` 툴을 호출하여 사용자의 승인을 기다립니다.
승인 전까지 코드를 수정하지 않습니다.
